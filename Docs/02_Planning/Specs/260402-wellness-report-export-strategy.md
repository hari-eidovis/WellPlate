# Strategy: Wellness Report Export (PDF + CSV)

**Date**: 2026-04-02
**Source**: `Docs/01_Brainstorming/260402-feature-prioritization-from-deep-research-brainstorm.md` (Feature 1C)
**Status**: Ready for Planning

---

## Chosen Approach

**SwiftUI `ImageRenderer` → PNG share + CSV via `ShareLink`.**

Build a `WellnessReportView` (a self-contained SwiftUI view that lays out a 7-day wellness summary card), render it to a PNG image using `ImageRenderer` (iOS 16+, available since we target iOS 26), and expose both formats via `ShareLink` in a single share sheet. Wire into the **already-existing stub** in `ProgressInsightsView` — `showShareSheet` is declared and the share button is already present, but nothing is connected to it.

---

## Rationale

- **`ImageRenderer` over PDFKit/UIPrintPageRenderer**: True multi-page PDF is over-engineered for an MVP. A rendered image card is immediately shareable to Messages, Instagram, saved to Photos, and looks polished. Users can screenshot PDFs anyway — a beautiful image card is higher perceived value. If full PDF is needed later, the same `WellnessReportView` can be fed to `UIPrintPageRenderer`.
- **`ShareLink` over `UIActivityViewController`**: Native SwiftUI API, no UIKit wrapper needed. Supports sharing multiple items (image + CSV file) in one call on iOS 16+.
- **Existing `showShareSheet` stub over a new entry point**: Reduces UI changes to near zero — the share button already exists in `ProgressInsightsView`'s custom nav bar (line 188). We just need to attach a sheet to it.
- **7-day window**: Weekly cadence matches the research finding that users export for therapy/clinician workflows (weekly check-ins). The existing `ProgressInsightsView` already has `.week` as the default `TimeRange`.

---

## Affected Files & Components

- `WellPlate/Features + UI/Progress/Views/ProgressInsightsView.swift` — attach `.sheet(isPresented: $showShareSheet)` presenting the share sheet; pass 7-day data to the generator
- `WellPlate/Features + UI/Progress/Views/WellnessReportView.swift` *(new)* — self-contained SwiftUI card layout for rendering; not user-navigable, render-only
- `WellPlate/Features + UI/Progress/Services/WellnessReportGenerator.swift` *(new)* — renders `WellnessReportView` to PNG via `ImageRenderer`; generates CSV string; returns `(UIImage, Data)` tuple
- `WellPlate/Features + UI/Progress/Views/WellnessReportShareSheet.swift` *(new)* — SwiftUI sheet that shows a preview of the report card + `ShareLink` buttons for image and CSV

Data sources already accessible in `ProgressInsightsView`:
- `allFoodLogs: [FoodLogEntry]` — via existing `@Query`
- `UserGoals` — via existing `@Query`
- `StressReading` — add `@Query` to `ProgressInsightsView`
- `WellnessDayLog` — add `@Query` to `ProgressInsightsView`

---

## Architectural Direction

```
ProgressInsightsView
  └── .sheet(isPresented: $showShareSheet)
        └── WellnessReportShareSheet
              ├── WellnessReportView (preview, live SwiftUI)
              └── ShareLink buttons
                    ├── Image (PNG) — rendered by WellnessReportGenerator
                    └── CSV file — generated by WellnessReportGenerator
```

`WellnessReportGenerator` is a plain `struct` with two static methods:
- `renderImage(from view: WellnessReportData) async -> UIImage`
- `generateCSV(from data: WellnessReportData) -> Data`

`WellnessReportData` is a plain `struct` (not a SwiftData model) holding the 7-day summary already computed from `ProgressInsightsView`'s existing `filteredData` / `currentPeriodStats`.

---

## Report Content (7-day card)

1. **Header**: App name + date range + "Weekly Wellness Report"
2. **Stress**: Avg score, trend arrow, best/worst day
3. **Nutrition**: Avg calories vs goal, top macro compliance, meal count
4. **Activity**: Avg steps, active energy, exercise minutes
5. **Mood**: Most common mood emoji for the week
6. **Hydration**: Avg water glasses vs goal
7. **Footer**: "Generated by WellPlate · Data stays on your iPhone"

CSV columns: `date, stress_score, calories, protein_g, carbs_g, fat_g, steps, water_glasses, mood, exercise_min`

## Design Constraints

- `WellnessReportView` must be a fixed-width layout (390pt — iPhone 15 Pro width) for consistent `ImageRenderer` output regardless of device
- Use `AppColors` and `.r()` font extension for brand consistency
- `WellnessReportShareSheet` must use `.presentationDetents([.large])` — the preview card needs space
- No new `@Model` classes or SwiftData migrations
- `WellnessReportGenerator` must be callable from `@MainActor` (ImageRenderer requires main actor)
- Follow existing `ProgressInsightsView` color scheme (it has its own warm gradient — match it in the report card)

---

## Non-Goals

- No multi-page PDF in this iteration
- No email integration (ShareLink covers all share targets)
- No 30-day or 90-day report variants (weekly only for MVP)
- No server-side processing or cloud upload
- No changes to `StressInsightService` or `HealthKitService`

---

## Open Risks

- **`ImageRenderer` scale on high-DPI**: Default `@2x` scale produces large files. Use `scale: 2.0` explicitly and compress PNG output with `UIImage.jpegData(compressionQuality: 0.85)` for reasonable file size (~300–500KB). Risk: Low.
- **SwiftData query in sheet**: `ProgressInsightsView` needs `StressReading` and `WellnessDayLog` queries added. These are simple 7-day filtered queries identical to existing `FoodLogEntry` pattern. Risk: Low.
- **`WellnessReportData` construction performance**: Aggregating 7 days of data on main thread. Given the small window (7 × few records each), this is fast. Risk: Very Low.
