# Brainstorm: Journal / Gratitude Prompts Tied to Mood

**Date**: 2026-04-07
**Status**: Ready for Planning
**Roadmap**: F4 — Phase 2 (Engagement Layer)

## Problem Statement

WellPlate captures mood as a 5-emoji check-in and stores it as an integer in `WellnessDayLog.moodRaw`. This is quantitative but shallow — users select an emoji and move on. There's no space for *why* they feel that way, no reflection habit, and no longitudinal awareness of emotional themes. The daily loop currently lacks a "pull" mechanism that makes users *want* to open the app beyond logging duties.

Mood trackers succeed via a low-friction reflection habit. Apple's HealthKit State of Mind APIs are explicitly designed around reflection, and Apple frames the practice as beneficial. Foundation Models on iOS 26 enable privacy-preserving, on-device prompt generation — no server, no data leak.

## Core Requirements

- Daily text journal entry (one or more per day)
- AI-generated prompts based on current mood, stress signals, and recent patterns
- Themes-over-time summary view ("what keeps coming up")
- Privacy-first: all journal text stays on-device (SwiftData), AI runs on-device (Foundation Models)
- Low friction: <30 seconds to complete a prompted entry
- Graceful degradation when Foundation Models unavailable (pre-iOS 26, unsupported hardware)

## Constraints

- **No backend**: All journal data persisted locally via SwiftData
- **Foundation Models availability**: iOS 26+ only, may not be available on all devices → must have template fallback
- **Tab real estate**: Currently 3 tabs (Home, Stress, Profile). Profile is a placeholder
- **Existing patterns**: Must follow MVVM + Service Layer, `@MainActor` isolation, `.r()` fonts, `.appShadow()`, card styling
- **SwiftData ModelContainer**: New model must be registered in `WellPlateApp.swift`
- **Single `.sheet(item:)`**: Navigation via sheet enums, not stacked `.sheet()` calls

## Approach 1: Journal as Post-Mood Flow (Inline Extension)

**Summary**: After the user taps a mood emoji on the Home screen, a journal prompt slides up below the confirmed mood card. The journal is a natural continuation of mood check-in, not a separate destination.

### UX Flow
1. User selects mood emoji on `MoodCheckInCard` → confirmation animation
2. Below the confirmed mood pill, a "How are you feeling?" text field appears with an AI-generated prompt
3. User can type (or skip), then tap "Done"
4. Entry saved as `JournalEntry` linked to today's `WellnessDayLog`
5. Past entries accessible via a "Journal" button in the mood confirmation area or a dedicated view

### Prompts
- Foundation Models generates a prompt from: today's mood, yesterday's stress score, recent sleep quality, recent food patterns
- Fallback: curated prompt library (~50 prompts) categorized by mood level and stress context

### Themes View
- Accessible from Home or Profile
- Shows word cloud / tag frequency from past entries
- Foundation Models summarizes recurring themes weekly

**Pros**:
- Zero navigation friction — journal appears exactly when user is in reflection mindset
- Builds on existing `MoodCheckInCard` with minimal new UI surface
- Highest chance of adoption (path of least resistance)
- Feels like a natural extension, not a bolted-on feature

**Cons**:
- Limited space on Home screen for longer entries
- Can't easily browse/write entries outside mood flow
- Couples journal to mood timing — what if user wants to journal at a different time?
- Home screen already has several cards; adding more increases scroll depth

**Complexity**: Low
**Risk**: Low (minimal new infrastructure)

## Approach 2: Dedicated Journal Tab (Replace Profile Placeholder)

**Summary**: Replace the Profile placeholder (tab 2) with a full Journal experience. Profile moves to a settings gear in Home. The Journal tab becomes the daily reflection hub.

### UX Flow
1. Tab bar: Home | Stress | Journal (book icon)
2. Journal tab opens to today's entry (or empty prompt card)
3. Top: today's mood pill (read-only, set from Home) + stress score badge
4. Center: AI-generated prompt card with text editor
5. Below: scrollable past entries (card per day, expandable)
6. Toolbar: calendar picker, themes/insights button

### Prompts
- Same Foundation Models approach as Approach 1
- Richer context: can show 2-3 prompt options, user picks one
- "Surprise me" shuffle button for prompt variety

### Themes View
- Dedicated section within Journal tab
- Monthly theme cards generated by Foundation Models
- Timeline view showing mood + journal entry pairs

**Pros**:
- Full-screen dedicated space for reflection
- Can handle long-form entries comfortably
- Natural home for themes/insights view
- Profile placeholder is unused — journal is higher value
- Clear information architecture: Home (log) → Stress (analyze) → Journal (reflect)

**Cons**:
- Journal without today's mood feels incomplete — dependency on Home flow
- Profile features will eventually need a home (settings gear pattern works)
- Heavier implementation (full tab, not just a card)
- Users must switch tabs to journal — adds friction vs. inline approach

**Complexity**: Medium
**Risk**: Low–Medium (tab restructure is straightforward but touches MainTabView)

## Approach 3: Hybrid — Inline Prompt + Journal Sheet

**Summary**: Quick journal prompt appears inline after mood check-in (like Approach 1), but tapping "Write more" or a journal icon opens a full sheet/view for deeper reflection. Past entries and themes live in a sheet accessible from Home or Stress.

### UX Flow
1. Mood check-in on Home → inline 1-2 line prompt appears
2. User can quick-respond (short text) or tap "Write more →" to open full journal sheet
3. Full sheet: rich text area, AI prompt at top, mood/stress context sidebar
4. "Journal History" accessible via toolbar button (book icon) → shows past entries + themes
5. Optional: journal icon in Stress tab toolbar for cross-feature access

### Prompts
- Quick prompt: single short prompt (1 sentence) for inline
- Full sheet: 2-3 prompt options + free-form
- Foundation Models with template fallback

### Themes View
- Separate sheet/view (not inline)
- Foundation Models weekly summary
- Mood correlation: "When you wrote about [work], your mood was usually [okay/bad]"

**Pros**:
- Best of both worlds: low friction for quick entries, depth for when users want it
- Doesn't require tab restructure
- Progressive disclosure: simple → detailed
- Journal history is a sheet, not a tab — less architectural change
- Users who skip the inline prompt still see the option without being pushed

**Cons**:
- Two UI surfaces to maintain (inline + sheet)
- Sheet navigation needs careful enum design (new cases in existing sheet enums, or new sheet state)
- Inline prompt disappears after mood is logged — need a way to access journal later in the day

**Complexity**: Medium
**Risk**: Low

## Approach 4: Journal as Stress Feature Extension

**Summary**: Journal lives inside the Stress tab as a new card/section. Rationale: journaling is a stress intervention, and Stress tab is already the "understand yourself" hub.

### UX Flow
1. Stress tab gets a "Daily Reflection" card (similar to existing factor cards)
2. Tapping opens a journal sheet (via `StressSheet` enum)
3. AI prompts contextualized with stress data (score, factors, vitals)
4. Themes view shows stress-journal correlations

**Pros**:
- Thematic fit: reflection as stress management
- Reuses existing `StressSheet` pattern
- Stress tab is the analytical hub — journal adds qualitative data to quantitative scores
- No tab restructure needed

**Cons**:
- Journal is broader than stress — confining it feels limiting
- Stress tab already has many cards; adding another increases density
- Mood check-in is on Home, not Stress — journal disconnected from mood flow
- Users who don't engage with Stress tab won't discover journaling

**Complexity**: Low–Medium
**Risk**: Medium (feature-scope mismatch could hurt adoption)

## Edge Cases to Consider

- [ ] **Empty state**: First-ever journal entry — need compelling onboarding prompt
- [ ] **Mood not logged yet**: If user opens journal before mood check-in, should we prompt mood first or allow journal without mood context?
- [ ] **Multiple entries per day**: Allow or enforce one-per-day? One-per-day is simpler but users may want morning/evening entries
- [ ] **Entry length limits**: Foundation Models context window for theme analysis; SwiftData storage for very long entries
- [ ] **Foundation Models unavailable**: Template prompts must be high quality, not obviously canned
- [ ] **Data migration**: If user upgrades to iOS 26 mid-use, historical entries should still work
- [ ] **Export**: Journal entries should be included in wellness reports (clinician export story)
- [ ] **Deletion**: User should be able to delete individual entries; consider "delete all journal data" option
- [ ] **Search**: As entries accumulate, users will want to search. When to add search?
- [ ] **Prompt fatigue**: Same prompt types get stale. Need enough variety and context-sensitivity
- [ ] **Offline**: Foundation Models runs on-device — no connectivity concern, but model download state matters
- [ ] **Dark mode**: Text editor styling must adapt (already handled by `.systemBackground` pattern)
- [ ] **Accessibility**: VoiceOver for prompts, text editor; Dynamic Type for journal cards
- [ ] **Streaks**: Journal streak as engagement mechanic? (Food logging already has streaks)

## Open Questions

- [ ] Should journal entries be queryable by Foundation Models for future insights (e.g., "looking back at your last month...")?
- [ ] Should prompts reference specific data points ("You slept 4.5 hours last night — how are you feeling about that?")?
- [ ] What's the minimum viable themes view? Word frequency? AI summary? Simple list of entries?
- [ ] Should the journal be included in the stress score calculation (sentiment analysis)?
- [ ] How many template fallback prompts are needed for reasonable variety? (~50? ~100?)
- [ ] Should there be guided journal "modes" (gratitude, stress debrief, evening wind-down)?

## Recommendation

**Approach 3: Hybrid — Inline Prompt + Journal Sheet** is the strongest choice.

### Rationale:
1. **Lowest friction entry point**: Inline prompt after mood check-in catches users at peak reflection mindset
2. **Progressive disclosure**: Quick entries for most days, deep reflection when they want it
3. **No tab restructure**: Avoids touching `MainTabView` — saves that real estate for future features or when Profile needs a real implementation
4. **Reuses patterns**: Sheet presentation follows existing `StressSheet` enum style; inline card follows `MoodCheckInCard` extension pattern
5. **Foundation Models pattern proven**: `StressInsightService` already demonstrates the exact generate-with-fallback pattern needed
6. **Themes view as separate sheet**: Keeps it discoverable without cluttering the main scroll

### Key design decisions for planning:
- **One entry per day** for MVP (simplicity; multi-entry can come later)
- **Foundation Models prompts + 50-prompt template library** for fallback
- **Journal accessible from Home** (inline + toolbar icon for history)
- **Themes view**: Foundation Models weekly summary with template fallback showing top recurring words/phrases
- **SwiftData `JournalEntry` model**: date, text, mood at time of writing, prompt used, optional stress score
- **Export integration**: Include journal entries in `WellnessReportGenerator` output
